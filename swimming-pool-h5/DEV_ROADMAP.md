## 🎯 开发优先级规划（基于业务价值）

### **阶段一：核心业务闭环（P0 - MVP）**

> **目标：** 实现完整的预约-支付-使用流程，满足基本业务需求

#### 1.1 预约模块 ⭐⭐⭐⭐⭐

**优先级：** **P0（最高）**
**开发周期：** 3-4天
**业务价值：** 系统核心功能，用户主要使用场景

##### 功能清单

| 功能 | 说明 | 后端API | 前端组件 | 复杂度 |
|------|------|---------|----------|--------|
| **泳道展示** | 展示所有泳道及状态 | GET /h5/lanes/list | 泳道网格组件 | ⭐⭐ |
| **时段选择器** | 日期+时段选择（日历控件） | GET /h5/timeslots/available | 日历组件 | ⭐⭐⭐ |
| **实时余位** | 显示各泳道剩余名额 | GET /h5/timeslots/availability | 进度条组件 | ⭐⭐ |
| **发起预约** | 选择时段后提交预约 | POST /h5/reservation/create | 确认弹窗 | ⭐⭐ |
| **预约冲突检测** | 动态时间片锁机制 | - | Redis分布式锁 | ⭐⭐⭐⭐ |
| **预约成功页** | 显示预约详情和倒计时 | - | 结果页组件 | ⭐⭐ |

**核心创新点：**
- 🔥 动态时间片粒度：高峰期15分钟/低峰期30分钟
- 🔥 并发冲突控制：Redisson分布式锁确保300人并发预约冲突率≤0.5%

**技术挑战：**
- 时段选择器UI复杂（需要支持拖拽选择）
- 实时余位计算性能优化
- 分布式锁的正确使用

**验收标准：**
- [ ] 能查看未来7天的可预约时段
- [ ] 能看到每个泳道的实时剩余名额
- [ ] 300人同时预约不产生冲突
- [ ] 预约成功后发送邮件通知

#### 1.2 订单模块 ⭐⭐⭐⭐

**优先级：** **P0（核心业务）**
**开发周期：** 2-3天
**依赖：** 预约模块完成后

##### 功能清单

| 功能 | 说明 | 后端API | 前端组件 | 复杂度 |
|------|------|---------|----------|--------|
| **订单列表** | 显示所有订单（状态筛选） | GET /h5/orders/list | 订单卡片列表 | ⭐⭐ |
| **订单详情** | 查看订单完整信息 | GET /h5/orders/{id} | 详情页组件 | ⭐⭐ |
| **待支付订单** | 15分钟倒计时 | - | 倒计时组件 | ⭐⭐⭐ |
| **支付功能** | 支付宝/余额支付 | POST /h5/payment/pay | 支付弹窗 | ⭐⭐⭐ |
| **支付状态轮询** | 轮询支付结果 | - | 定时器 | ⭐⭐ |
| **取消订单** | 未支付时取消 | POST /h5/orders/cancel | 确认弹窗 | ⭐ |
| **退款申请** | 已支付订单申请退款 | POST /h5/refund/apply | 申请表单 | ⭐⭐ |

**订单状态流转：**
```
待支付(15min) → 已支付 → 已完成
     ↓           ↓
  已取消      已退款
```

**技术挑战：**
- 支付15分钟倒计时（考虑关闭页面后继续计时）
- 支付结果异步通知处理
- 支付宝沙箱环境测试

**验收标准：**
- [ ] 订单列表支持按状态筛选
- [ ] 15分钟未支付自动取消
- [ ] 支付成功后邮件通知
- [ ] 支持退款申请和审核流程
